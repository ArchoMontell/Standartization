# Контрольні питання ЛР-8

## 1. У чому полягає принцип Documentation as Code?

**Documentation as Code (DaC)** — практика управління документацією як частиною кодової бази.

**Принципи**:
- Документація у текстових форматах (Markdown, AsciiDoc)
- Версіонування в Git разом із кодом
- Code Review для документації
- Автоматична генерація сайтів (MkDocs, Sphinx)
- Посилання з кодом та вимогами

**Переваги**:
- Документація завжди актуальна (в тій же версії, що й код)
- Історія змін видна в Git
- Легко колаборувати (PR, review, merge)
- Можна автоматизувати (CI/CD pipeline)

---

## 2. Що таке Single Source of Truth (SSOT)?

**SSOT** — один офіційний документ для кожного аспекту системи.

| Компонент | SSOT | Залежні документи |
|-----------|------|------------------|
| Вимоги | SSD | Test Strategy, Traceability Matrix |
| Архітектура | SDD | Implementation guides, API docs |
| Інфраструктура | ISD | DevOps runbooks |
| Тестування | Test Strategy | Test cases, test reports |

**Чому важливо**:
- Уникаємо дублювання й конфліктів
- При зміні оновлюємо одне місце
- Все залежить від офіційного джерела
- Єдине розуміння у команді

---

## 3. Чому Markdown є базовим форматом для документації?

| Аспект | Markdown | Word/PDF |
|--------|----------|----------|
| **Читаність** | Текст, структура видна | Форматування складне |
| **Git友好** | Diff видно чітко | Бінарні файли, конфлікти |
| **Редагування** | Простий текстовий редактор | Потрібна спеціальна програма |
| **Генерація** | MkDocs, Sphinx → HTML, PDF | Ручна конвертація |
| **Версіонування** | Ефективно в Git | Великі різниці, складно мергити |

**Результат**: Markdown = документація як код, легко підтримувати.

---

## 4. Як забезпечується актуальність документації у DaC-підході?

**1. Версіонування разом із кодом**
- Кожна версія коду = версія документації
- При checkout коду виводиться правильна документація

**2. Pull Request процес**
- Зміни коду + оновлення документації = один PR
- Code Review включає перевірку документації

**3. Autоматизація (CI/CD)**
- При зміні документації генерується новий сайт
- Тести можуть перевіряти посилання, синтаксис

**4. SSOT принцип**
- Одне місце для кожного фактула
- Залежні документи посилаються на SSOT

**5. Культура команди**
- Розробники оновлюють документацію як обов'язок
- Сеньйори перевіряють документацію в PR

**Результат**: Документація ніколи не відстає від кодуу, завжди актуальна й надійна.
