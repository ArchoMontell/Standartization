# Definition of Done (DoD) для документації

Цей документ формулює критерії "Definition of Done" (DoD) для інженерних змін у проекті з точки зору документаційної відповідності. Жодна зміна (Pull Request) не може бути схвалена та прийнята в `main`, якщо не дотримано цього DoD.

## Типи змін та обов'язкові дії

| Тип інженерної зміни | Обов’язкові дії з документацією | Автоматизована перевірка (CI) |
| --- | --- | --- |
| **Додавання нового API-ендпоінта** | Оновити файл `openapi.yaml` (додати path, методи, request/response, schemas). | OpenAPI spectral validation. |
| **Зміна формату відповіді існуючого API**| Оновити схему (`components/schemas`) в `openapi.yaml`. | OpenAPI spectral validation. |
| **Зміна логіки, що зумовлює Breaking changes в API** | Вказати необхідність переходу до нової мажорної версії API; оновити `info.version` та URL-шляхи; оновити Migration Guide у MkDocs. | OpenAPI linting, перевірка версіонування у pipeline. |
| **Додано новий UI-компонент** | Розробити Storybook стейтменти (мінімум Default, Error, Loading). Описати у `docs/storybook.md`. | Збірка Storybook UI (`build-storybook`) проходить без помилок. |
| **Нефункціональні зміни / Рефакторинг** | Зміни документації, як правило, не вимагаються, окрім випадку, коли зміни впливають на продуктивність чи системні обмеження (оновити `overview.md` за потреби). | Успішна перевірка існуючої документації на відсутність зламів (MkDocs build pass). |

## Критерії готовності (Загальні для всіх PR)

Щоб завдання вважалося DONE, повинні бути виконані наступні критерії щодо документації:
1. **Існування**: Валідний YAML-файл для OpenAPI та описи Storybook існують і відображають останні зміни.
2. **Формат**: OpenAPI валідний та відповідає специфікації версії 3.0+.
3. **Місце перевірки**: CI pipeline успішно виконує валідацію (`Spectral`) і збірку (`MkDocs`, `Storybook`), про що свідчить зелений чекмарк в Pull Request. Крім того, наявність і правильність оновленої документації перевіряється Reviewer-ом перед merge-коммітом.
