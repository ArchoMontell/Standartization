# Контрольні питання ЛР-8

## 1. Чим SSD відрізняється від SDD?

| Параметр | SSD | SDD |
|----------|-----|-----|
| **Що описує** | **ЩО** робить система | **ЯК** реалізована система |
| **Рівень** | Бізнес-вимоги | Технічна архітектура |
| **Аудиторія** | Замовник, QA | Розробники, архітектори |
| **Вміст** | Функціональні/нефункціональні вимоги | Компоненти, API, БД, технологічний стек |
| **SSOT** | Джерело істини для вимог | Джерело істини для реалізації |

**Приклад**: SSD каже "система має 99.9% доступності", SDD каже "використовуємо Kubernetes для масштабування".

---

## 2. Яка роль ISD у життєвому циклі ПЗ?

**ISD** — документ про інфраструктуру, потрібен на кожному етапі:

- **Планування**: Вибір серверів, хмари, вартості
- **Розроблення**: Docker, CI/CD pipeline, середовища
- **Тестування**: Подібність production-середовища
- **Розгортання**: Інструкції для DevOps
- **Експлуатація**: Моніторинг, резервування, відновлення після збоїв

**Без ISD**: Система не розгортається, не масштабується, падає на production.

## 3. Навіщо потрібна матриця простежуваності?

**Matриця простежуваності** — таблиця вимога ↔ тест.

| Вимога | Тест | Статус |
|--------|------|--------|
| FR-01: Реєстрація | TC-01: Успішна реєстрація |
| FR-02: Пошук рейсів | TC-02: Пошук за датою |
| NFR-01: ≤300 мс | TC-03: Тест навантаження |

**Навіщо**:
- Гарантує, що кожна вимога протестована
- Легко знайти тести при зміні вимог
- Доказ якості для аудиту
- Уникаємо невикористаних функцій або невкритих вимог

## 4. Чому документація впливає на якість продукту?

| Без документації | З документацією |
|------------------|-----------------|
| Розробники угадують вимоги → баги | Чітка SSD → правильна реалізація |
| QA не знає, що тестувати → неповне тестування | Test Strategy → повне тестування |
| Команда розумує по-різному → конфлікти | SSOT → єдине розуміння |
| Баги знаходяться на production | Баги знаходяться на dev |

**Висновок**: Документація = шаблон для якості. Без неї система буває хаотична, з похибками, недовірчива.

**Documentation as Code** гарантує, що документація завжди актуальна разом із кодом.
