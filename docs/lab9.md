# Лабораторна робота №9

## Контрольні питання

**1. Чому API-документація є контрактом?**
Тому що вона чітко визначає правила гри між клієнтом та сервером (формати запитів/відповідей, типи даних, маршрути). Обидві сторони (наприклад, Frontend та Backend команди) погоджуються дотримуватися цих правил, що гарантує сумісність їхніх компонентів та дозволяє вести паралельну незалежну розробку.

**2. У чому перевага API-first?**
Підхід API-first полягає у тому, що спочатку проектується інтерфейс (API), і лише потім пишеться код. Основні переваги: можливість паралельної роботи команд (клієнт може використовувати мок-сервер, поки бекенд ще розробляється), рання валідація вимог (можна знайти архітектурні недоліки до написання коду), та краща якість фінального API (він проектується під потреби клієнта, а не як відображення бази даних).

**3. Яку роль відіграє Swagger UI?**
Swagger UI автоматично генерує інтерактивну візуальну веб-сторінку на основі OpenAPI/Swagger специфікації. Його головна роль — надати зручний інтерфейс для перегляду документації людьми (розробниками, тестувальниками) та дозволити виконувати тестові запити безпосередньо з браузера без спеціальних програм на кшталт Postman.

**4. Навіщо потрібен Storybook?**
Storybook — це інструмент для розробки та тестування UI-компонентів ізольовано від основної бізнес-логіки застосунку. Він слугує як "каталог" або візуальна документація для frontend-розробників та дизайнерів, дозволяючи переглядати компоненти у різних станах (default, error, loading) і забезпечує узгодженість дизайну по всьому проекту.

**5. Чому документація має бути частиною Definition of Done?**
Завдяки цьому ми гарантуємо, що документація ніколи не застаріє. Якщо критерій "Документація оновлена" є в DoD, то розробник фізично не зможе закрити задачу (або злити Pull Request), поки не оновить OpenAPI, Storybook чи текстові пояснення. Це забезпечує принцип Single Source of Truth та постійну актуальність контрактів.
